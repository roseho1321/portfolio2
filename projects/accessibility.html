Accessibility @ FMI

Design Systems

2022

I helped redesign FMI’s design system to align with WCAG guidelines, whilst reducing redundancy in components. I learned, while simultaneously educated my peers, about core inclusive design principles and created design systems documentation for product teams to apply to their custom solutions for diverse users.

*Some content cannot be shown due to NDA

Research and Context

Foundation Medicine is a company that provides genomic testing to cancer patients to match them with treatments and clinical trials. A partnership with the Departmetn of Veterans Affairs prompted the start of the accessibility project. To get this started, we did an accessibility audit with a third party to get a better idea of what we needed to work on.

Since we were designing for somewhat of a niche audience, primarily health care providers, we really wanted to keep in mind the types of disabilities to ensure that we were being sufficiently broad with our definition of accessibility.

  • Multiple versions of the same component with inconsistent spacing, color, text sizes, creating additional work for development teams: Simplify components for a strong, clear, visual language.
  • Lack of documentation results in confusion when creating new designs and components. It is unclear what the appropriate context for each color, text style, etc: Create structured documentation.
  • While files were migrated from Sketch to Figma, we did not take full advantage of Figma’s features, resulting in inefficient use of components: Use variants and component properties to create an efficient component library.
  • Confusing file organization results in additional manual communication: Reorganize files for intuitive organization

Next, we gathered pain points from the design team. the biggest pain point was that there were multiple versions of the same component that were all slightly different and using different naming conventions, meaning additional work for design and dev, and a more confusing user experience. Other pain points were lack of documentation, not enough use of variants and components properties, and confusing file organization. We proposed high level solutions that also became the scope of the project.

To solve the multiple components issue, I put together a component directory of all of the components across the different teams. And this was really eye opening seeing all the variants side by side and inconsistencies with text size, spacing, alignment, color, etc. Putting together the component directory also helped with standardized the naming conventions for components.

Creating a Design System

At a high level, we wanted this to be sustainable, recognizable and cohesive. We didn’t want to redesign the brand, we just wanted to create a system that we could continue to build from and strengthen.

Then, we created a nested structure so people could easily find what they’re looking for.

  • Atoms: high-level system patterns, i.e. color, typography, shapes, etc.
  • Molecules: parts of the system that cannot be further broken down, i.e. control inputs, labels, etc.
  • Components: containers of system patterns, i.e. cards, modules, etc.

Next, we created text styles that would pass text size guidelines for accessibility, which was the second biggest bucket of issues that were found in the audit.

Documentation

For documentation, we started high level documentation for the components that would be used across teams, as well as color usage. We also developed the structure for a longer term solution that would be a more thorough documentation of each component as collaboration between design and engineering.

Learnings and Impact

These accessibility efforts led to a successful partnership with Veterans Affairs, as well as better communication with developers and within the design team.

As we continue to expand our offerings, it’s important to realize that the work is never done and there are also new biases to uncover.
